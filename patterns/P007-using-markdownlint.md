# Using markdownlint-cli2

markdownlint-cli2 is the Markdown linter, pinned exactly as a devDependency and configured via a committed `.markdownlint-cli2.jsonc`. The config enables all default rules, disables specific rules ONLY with an inline comment explaining why, and lists ignored paths. It's exposed as a `lint:md` script and composed into the project's `lint` alongside the other linters. Markdownlint is the markdown arm that Biome (P005) does not cover.

## When to use

- Any project that ships Markdown (README, docs/, AGENTS.md, RFCs) and wants it linted in CI.
- Any project already following P005 (Biome) — Biome owns JS/TS; markdownlint-cli2 owns `.md`. The two are complementary, not alternatives.
- Any project where the lint pass must be a single command (`pnpm lint`) that fans out to every file type.

## How it works

### Pin markdownlint-cli2 exactly as a devDependency

`markdownlint-cli2` is a devDependency, pinned to an exact version with no caret. Take the latest stable release and pin it:

```json title="package.json"
{
  "devDependencies": {
    "markdownlint-cli2": "0.17.2"
  }
}
```

```json title="package.json"
// Wrong — a caret lets the installed version drift between machines and CI
{
  "devDependencies": {
    "markdownlint-cli2": "^0.17.2"
  }
}
```

markdownlint rules can change between versions (new rules added, defaults tweaked); an exact pin plus the committed lockfile (P004) makes the lint result reproducible.

### Commit a .markdownlint-cli2.jsonc — JSONC, not JSON

The config file lives at the repo root. Use the `.jsonc` extension deliberately: it allows `//` comments, which is what lets you annotate every disabled rule with the reason. That annotation is the whole point of the JSONC choice — a disable without a rationale is a gap.

```jsonc title=".markdownlint-cli2.jsonc"
{
  "config": {
    "default": true,
    // long lines are common in README tables and code blocks
    "MD013": false,
    // inline HTML is used in the README logo
    "MD033": false,
    // README/AGENTS don't start with a heading (logo image / redirect)
    "MD041": false
  },
  "ignores": ["node_modules/", "CHANGELOG.md", ".claude/", ".harness/"]
}
```

### Enable all defaults, then opt out surgically

`config.default: true` turns on the full default rule set. Individual rules are turned off (`false`) only when the project legitimately needs to — and each one carries an inline comment naming the concrete reason:

| Rule | What it checks | Why harness disables it |
| --- | --- | --- |
| MD013 | Line length | Long lines are common in README tables and fenced code blocks |
| MD033 | Inline HTML | The README logo uses an `<img>` / `<a>` tag |
| MD041 | First line must be a heading | README/AGENTS start with a logo image or a redirect, not a heading |

Do not disable a rule just to make a build pass, and do not copy these opt-outs into a project that doesn't share the reason. Each `false` must be justified by something specific to THAT repo.

### Ignore generated, vendored, and auto-managed files

The `ignores` array excludes paths that shouldn't be linted:

```jsonc title=".markdownlint-cli2.jsonc"
{
  "ignores": ["node_modules/", "CHANGELOG.md", ".claude/", ".harness/"]
}
```

- `node_modules/` — dependencies (mandatory given P004).
- `CHANGELOG.md` — auto-generated by release tooling; linting it produces noise.
- `.claude/`, `.harness/` — tool-managed directories.

Add any other generated or vendored markdown (e.g. `dist/`, `site/`) the project has. The principle is the same as Biome's `files.ignore` (P005): never lint generated output.

### The lint:md script and the composed lint

markdownlint-cli2 is exposed as its own script and composed into the project's `lint`:

```json title="package.json"
{
  "scripts": {
    "lint": "pnpm lint:ts && pnpm lint:md && pnpm lint:sh && pnpm lint:docker && pnpm lint:actions",
    "lint:md": "markdownlint-cli2 \"**/*.md\" \"#node_modules\""
  }
}
```

- `lint:md` — runs markdownlint-cli2 over all Markdown. The glob `"**/*.md"` includes everything; `"#node_modules"` is the CLI's glob-negation syntax. (Note: `node_modules/` is ALSO in the config's `ignores`, so this is belt-and-suspenders. The config file is the canonical place for ignores — see anti-patterns.)
- `lint` — the single entry point that runs every linter in sequence. `lint:md` is one arm of it, alongside `lint:ts` (Biome, P005), `lint:sh` (shellcheck), `lint:docker` (hadolint), and `lint:actions` (actionlint).

The rule: there is ONE command — `pnpm lint` — and it fans out to every file type. Developers and agents never need to know the per-type script names.

### Gate lint in CI

`pnpm lint` (which includes `lint:md`) is a required CI step:

```yaml title=".github/workflows/lint.yml"
steps:
  - uses: actions/checkout@<sha> # v5
  - uses: pnpm/action-setup@<sha> # v6
  - uses: actions/setup-node@<sha> # v6
    with:
      node-version: 24
      cache: pnpm
  - run: pnpm install --frozen-lockfile
  - run: pnpm lint
```

The pnpm setup follows P004; actions are SHA-pinned per P002.

## The AGENTS.md section

```markdown title="AGENTS.md"
## Markdown Linting

This project lints Markdown with **markdownlint-cli2**. Configuration lives in `.markdownlint-cli2.jsonc`.

- Lint all Markdown: `pnpm lint:md`
- Lint everything (Markdown + all other file types): `pnpm lint`

The config enables all default rules (`default: true`) and disables a few specific ones — each with an inline comment explaining why. Do not disable a rule to silence a violation; either fix the Markdown or, if the disable is genuinely justified, add it with a comment naming the concrete reason. Generated/vendored Markdown (`CHANGELOG.md`, `.claude/`, `.harness/`) is in the `ignores` array — add generated paths there rather than letting them fail lint.
```

## Anti-patterns to avoid

- **Disabling a rule without an inline comment.** A `false` with no rationale is an unexplained hole. The JSONC format exists so every disable can say WHY. If you can't write a concrete reason, the rule shouldn't be off.
- **Copying another project's opt-outs blindly.** harness disables MD013/MD033/MD041 for README-specific reasons (logo, tables, first-line image). Those reasons don't transfer. Each disable must be justified by something in YOUR repo.
- **Pinning with a caret.** `"markdownlint-cli2": "^0.17.2"` drifts between machines and CI. Pin exactly.
- **Duplicating ignores between the config and the CLI args.** `node_modules/` appears both in `.markdownlint-cli2.jsonc` `ignores` AND as `"#node_modules"` in the script. That's harmless but redundant — the config file is the single source of truth for ignores. Prefer the config; don't rely on inline CLI negations.
- **Linting generated Markdown.** `CHANGELOG.md`, release notes, and compiled docs are generated; lint failures there are noise that trains people to ignore the linter. Put them in `ignores`.
- **Running markdownlint-cli2 outside the composed `pnpm lint`.** The convention is a single `pnpm lint` command that fans out to every file type. A separate ad-hoc invocation bypasses the rest of the lint pass.
- **Using Biome to lint Markdown.** Biome (P005) owns JS/TS. For Markdown, markdownlint-cli2 is the tool. Don't split Markdown linting across both or add a second Markdown linter.
- **A plain `.json` config instead of `.jsonc`.** Without comments you lose the ability to annotate disables, which defeats the main reason for a checked-in config. Use `.markdownlint-cli2.jsonc`.
